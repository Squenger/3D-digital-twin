# 3D Digital Twin: Pipeline Architecture

This document outlines the architecture, data flow, and specific utilities of each module within the 3D Digital Twin reconstruction pipeline. The system is designed sequentially, moving from raw 2D micro-CT images to binarized voxels, and finally to a fully constructed 3D geometric model (Mesh/B-Rep) along with porosity analysis.

## Execution Flow

The typical execution order mimics the data transformation process:
`TifLoader` -> `SliceProcessor` -> `ModelBuilder` -> `VolumeAnalyzer` (Post-Processing) -> `ModelExporter`

---

## 1. Data Ingestion: `TifLoader`
**File:** `tif_loader.py`
**Input:** Folder path containing raw `.tif` or `.tiff` images.
**Output:** List of 2D Numpy arrays (grayscale images).

**Utility:**
This module acts as the entry point. It scans the provided directory, filters for TIFF files, and ensures they are loaded sequentially by extracting the numerical index from the filenames (natural sorting). It normalizes images across varying bit depths and color formats into a standardized 8-bit (0-255) grayscale tensor format to avoid artificial artifacts during thresholding.

---

## 2. Image Processing & Binarization: `SliceProcessor`
**File:** `slice_processor.py`
**Input:** List of 2D grayscale Numpy arrays from the `TifLoader`.
**Output:** Filtered, cropped, and binarized 3D volume (stack of 2D binary masks).

**Utility:**
This module transforms the continuous raw data into a strictly binary representation (matter vs. void).
- **Auto-Cropping:** It identifies the empty boundaries (Z, Y, X margins) of the sample by monitoring the standard deviation of pixel intensities across slices. If the standard deviation drops below a threshold, the slice is considered empty and is safely cropped out to save memory and processing time.
- **Binarization:** Uses the multi-layer Otsu Thresholding algorithm separating the pixels into matter and background classes, effectively stripping away the surrounding air or container noise.

---

## 3. Geometric Reconstruction: `ModelBuilder`
**File:** `model_builder.py`
**Input:** Binarized 3D voxel volume from the `SliceProcessor`.
**Output:** A 3D surface mesh (`trimesh.Trimesh`).

**Utility:**
Converts strictly discrete 3D pixel grids (voxels) into continuous mathematical representations (vertices and polygonal faces).
- **Marching Cubes:** Utilizes the Marching Cubes algorithm to trace and extract a polygonal hull (surface contour) wrapped precisely around the binarized matter.
- **Component Filtering:** Identifies disconnected clusters of faces (floating artifacts or noise outside the main object) and removes them based on a controllable geometric threshold, ensuring a single, stable continuous surface. 

---

## 4. Analysis & Volume Profiling: `VolumeAnalyzer`
**File:** `post_processing.py`
**Input:** Binarized 3D voxel volume from the `ModelBuilder` or `SliceProcessor`.
**Output:** Porosity percentage, cross-sectional PNG images, and graph plots.

**Utility:**
Estimates precisely the internal empty voids inside the target. 
- **Porosity Calculation:** Employs 2D morphological hole filling to determine the theoretical "solid" envelope of the object. It compares the volume of the true matter against the volume of the "filled" matter to calculate the global porosity rate.
- **Cross-Sectioning:** Extracts physical mathematical slices through the X, Y, and Z planes (non-linearly using trigonometric bounds to capture high-interest variance at the object's boundaries) and exports them to disk for visual analysis.

---

## 5. File Export & Representation: `ModelExporter`
**File:** `model_exporter.py`
**Input:** The `trimesh.Trimesh` object generated by `ModelBuilder`.
**Output:** Files written to disk (`.stl`, `.obj`, `.glb`, or `.step`).

**Utility:**
Handles the serialization of internal memory geometries into standard industry formats.
- **Surface Modes:** Quickly writes mesh information to native formats optimized for triangulated graphics (e.g., STL).
- **Solid Modes (B-Rep):** Uses CAD wrappers (FreeCAD or pythonocc-core) to bridge the gap between polygonal meshes and continuous mathematical splines (NURBS). It constructs boundary representation faces and knits them into closed solids required by structural or fluid simulation engines.

---

## 6. Interaction & Execution Contexts

- **The Core Engine (`main.py`):** Acts as the central orchestrator routing data down the pipeline. Depending on the flags passed (via Argument Parser), it routes arrays between modules, handling both single runs and dataset batch processing automatically.
- **The GUI (`gui.py` & `interactive_windows.py`):** A Tkinter/CustomTkinter wrapper layered on top of the pipelines that injects user callbacks directly prior to critical steps (e.g., injecting a manual visual cropping step between `SliceProcessor` and `ModelBuilder`).
